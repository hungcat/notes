# Notes プロジェクト実装プラン

## 目的

このプロジェクトはメモの蓄積と閲覧を目的としています。今後は「メモ入力画面」から生成したメモをリポジトリに登録し、履歴管理できるようにします。

## シナリオ A: メモ入力画面をDevモード限定にし、公開先の GitHub Pages から隠す

### 内容
- `docs:dev` で動作するローカル開発環境にのみメモ入力ページを表示。
- GitHub Pages 上の公開サイトではナビゲーションに表示しない。
- 入力内容をローカルの `docs/memo` に Markdown ファイルとして書き出し、Git の `add` / `commit` / `push` を実行する。

### メリット
- 実装が最も簡単
- GitHub Pages だけでは実現できない書き込み操作をローカル側で完結できる
- 認証や機密情報管理の負担が小さい

### デメリット
- リモートの公開サイトからは直接入力・保存できない
- 開発環境に依存する運用になる

## シナリオ B: GitHub Pages 上にメモ入力画面を公開し、ホワイトリスト登録された GitHub アカウントのみ許可する

### 内容
- 公開サイト上にメモ入力ページを表示
- GitHub OAuth とアクセストークンを使って利用者を認証
- ホワイトリストに登録されたユーザーだけリポジトリ書き込みを許可する

### メリット
- どこからでもアクセス可能
- 実運用で入力画面を公開できる

### デメリット
- GitHub Pages 単体では実現が難しい
- OAuth／アクセストークン管理を含むセキュリティ設計が必要
- フロントエンドのみでトークンを扱うのは危険

## シナリオ C: 入力画面は公開したまま、実際の書き込みは別のバックエンドまたは GitHub Actions に委譲する

### 内容
1. 入力画面を GitHub Pages 上に置く
2. 入力内容をサーバーレス関数や GitHub Actions に送信
3. サーバー側でホワイトリストチェックし、GitHub API でファイルを追加・コミット・プッシュする

### メリット
- 静的サイトのまま運用可能
- 機密情報をバックエンド側で管理できる

### デメリット
- バックエンドやワークフローの追加運用が必要
- 実装と管理コストが上がる

## 推奨

まずは **シナリオ A** で進めるのが現実的です。静的な GitHub Pages サイトの構成を壊さずに、ローカル開発環境でのメモ登録機能を実装できるためです。

## 現状の実装方針

- `docs/input.vue` にメモ作成 UI を実装
- `docs/.vitepress/config.ts` に dev モードのときだけ入力ページをナビゲーションに表示
- `pnpm docs:dev` 実行時にのみ動作するローカル API を追加
- 保存先は `docs/memo/` 以下の Markdown ファイル
- 保存時に `git add` / `git commit` / `git push` を実行する

## 拡張性向上のための抽象化

将来の外部ファイル化や GitHub Actions 移譲を考慮して、ファイル操作と Git 操作を抽象化しました。

### 設定インターフェース
```typescript
export interface NoteRepositoryConfig {
    baseDir: string;
    memoDir: string;
    dateFormat: string;
    fileExtension: string;
    gitRemote?: string;
    gitBranch?: string;
}
```

### アダプターインターフェース
- `FileSystemAdapter`: ファイル操作を抽象化（テストや外部ストレージ対応）
- `GitAdapter`: Git 操作を抽象化（GitHub API や外部サービス対応）

### 移譲の容易さ
- **GitHub Actions 移譲例**: GitHub API を使用したリモート操作
- **外部ストレージ移譲例**: S3 や他のクラウドストレージへの保存
- **設定外部化**: 環境変数や設定ファイルからの設定読み込み

これにより、シナリオ B/C への移行が容易になります。
